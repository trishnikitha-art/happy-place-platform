# Generation Manifest Specification

**Status:** Draft
**Purpose:** Define machine-readable constitution as compiler input for PING code generation
**Context:** Constitution → IR → Code Generator → Runtime (LLVM-style compilation model)

---

## 0. Constitutional Primitives (Frozen)

**Exactly Four Business Primitives:**

1. **Identity** (Customer, Crew, Vendor)
2. **Mission** (Estimate, Job, Project)
3. **Evidence** (Photo, Video, Voice, PDF)
4. **Observation** (RoofDamage, WaterLeak, BrokenWindow)

**Everything Else Is Generated:**
- Authority (generated from mutation ownership)
- Projection (generated from event subscriptions)
- Capability (generated from operation contracts)
- Policy (generated from executable rules)
- Event (generated from state transitions)

**No First-Class Objects:**
- ❌ Estimate Repository
- ❌ Job Service
- ❌ Customer Manager
- ❌ Command Handlers
- ❌ Query Handlers

These are generated by PING.

---

## 1. Overview

The Generation Manifest is a first-class constitutional artifact that makes the Constitution executable. Instead of being documentation for contractors, it becomes the source that PING agents compile into the application.

**Architecture (LLVM-style Compilation):**
```
PING Kernel
        ↓
Constitution
        ↓
Generation Manifest (Compiler Input)
        ↓
Intermediate Representation (IR)
        ↓
Code Generator
        ↓
Runtime
```

**Compiler Pipeline:**
```
Manifest Validator
        ↓
Manifest → IR
        ↓
IR → Event Model Generator
        ↓
IR → Repository Generator
        ↓
IR → Capability Runtime
        ↓
IR → API & SDK Generator
```

**Target Flow:**
```
Constitution
    ↓
PING
    ↓
Generate (80-90%)
    ↓
Humans Review
    ↓
Deploy
```

## 2. Manifest Structure (Compiler Input)

### 2.1 Root Schema

```yaml
version: "1.0"
tenant_id: "tenant-001"
constitution:
  identities: {...}
  missions: {...}
  evidence: {...}
  observations: {...}
  capabilities: {...}
  policies: {...}
```

**Removed:**
- claims (collapsed into observations)
- facts (collapsed into observations)
- authorities (generated from mutation ownership)
- workflows (replaced by DAGs)

### 2.2 Versioning

```yaml
version: "MAJOR.MINOR.PATCH"
# MAJOR: Breaking changes to manifest structure
# MINOR: New fields, backward compatible
# PATCH: Bug fixes, documentation updates
```

## 4. Missions (State Machine Only)

### 4.1 Schema (State Transitions Only)

```yaml
missions:
  MissionName:
    version: "1.0"
    description: "Human-readable description"
    owns:
      - RelatedIdentity
      - RelatedMission
      - Evidence
    states:
      - name: StateName
        allowed_transitions:
          - to: TargetState
            requires: [policy_name]
    policies:
      - PolicyName
```

**Removed:**
- events (generated from state transitions)
- commands (generated from state transitions)
- state_machine (replaced by simple state definition)

**Generated by PING:**
- Commands (e.g., AcceptEstimateCommand, RejectEstimateCommand)
- Handlers
- Validators
- Tests
- Events (canonical: Transitioned)
- Replay logic
- Projections

### 4.2 Example: Estimate

```yaml
missions:
  Estimate:
    version: "1.0"
    description: "Estimate mission for customer property"
    owns:
      - Customer
      - Property
      - Tasks
      - Evidence
    states:
      - name: draft
        allowed_transitions:
          - to: pending
            requires: [MustHaveCustomer, MustHaveProperty, ValidAmount]
      - name: pending
        allowed_transitions:
          - to: accepted
            requires: []
          - to: rejected
            requires: []
          - to: expired
            requires: []
      - name: accepted
        allowed_transitions:
          - to: converted
            requires: []
      - name: rejected
        allowed_transitions: []
      - name: expired
        allowed_transitions: []
      - name: converted
        allowed_transitions: []
    policies:
      - MustHaveCustomer
      - MustHaveProperty
      - ValidAmount
```

**PING Generates:**
- CreateEstimateCommand
- AcceptEstimateCommand
- RejectEstimateCommand
- ExpireEstimateCommand
- ConvertToJobCommand
- All handlers
- All validators
- All tests
- Events: Transitioned (aggregate: Estimate, from: draft, to: pending)
- Replay logic
- Projections

## 7. Capabilities (ABI with Operations)

### 7.1 Schema (Operation ABI)

```yaml
capabilities:
  CapabilityName:
    version: "1.0"
    description: "Human-readable description"
    operations:
      - name: OperationName
        inputs:
          - name: input_name
            type: string | number | boolean | date | enum | array | object
            required: true | false
        outputs:
          - name: output_name
            type: string | number | boolean | date | enum | array | object
        errors:
          - error_name: error_description
        side_effects:
          - side_effect_description
        required_authorities:
          - authority_name
        replay_behavior: idempotent | replayable | non_replayable
        idempotency_key: field_name  # optional
    policies:
      - PolicyName
```

**ABI Components:**
- **Inputs:** Operation parameters with types
- **Outputs:** Operation return values with types
- **Errors:** Possible error conditions
- **Side Effects:** External effects (e.g., sending email, charging card)
- **Required Authorities:** Which authorities must authorize the operation
- **Replay Behavior:** How the operation behaves during replay
- **Idempotency Key:** Field that makes operation idempotent (optional)

## 8. Policies (Executable)

### 8.1 Schema (Executable Rules)

```yaml
policies:
  PolicyName:
    version: "1.0"
    description: "Human-readable description"
    enforcement: runtime | constitutional | hybrid
    scope: all | identity | mission | evidence | observation | capability
    rules:
      - requires:
          - entity: EntityName
        before:
          - event: EventName
        when:
          - condition: ConditionName
        assert:
          - assertion: AssertionName
```

**Executable Policy Components:**
- **requires:** Entities that must exist before operation
- **before:** Events that must occur before operation
- **when:** Conditions that must be true for operation
- **assert:** Assertions that must hold after operation

**PING Generates:**
- Validators
- Policy enforcement logic
- Policy violation events

## 9. Authorities (Generated from Mutation Ownership)

**Authorities are generated from mutation ownership defined in primitives.**

**No explicit authority definition needed.**

### 9.1 Generated Authority Schema

PING generates authorities based on what each primitive can mutate:

```yaml
# PING infers this from primitive definitions
MissionAuthority:
  can_mutate:
    - Mission State
    - Task Assignment
    - Crew Assignment
  cannot_mutate:
    - Evidence
    - Identity
    - Observation

ObservationAuthority:
  can_mutate:
    - Observation State
    - Assertion State
  cannot_mutate:
    - Mission
    - Identity
    - Evidence

EvidenceAuthority:
  can_mutate:
    - Evidence Metadata
    - Evidence Lineage
  cannot_mutate:
    - Mission
    - Identity
    - Observation

IdentityAuthority:
  can_mutate:
    - Identity Fields
    - External Identity Mappings
  cannot_mutate:
    - Mission
    - Evidence
    - Observation
```

**Benefits:**
- Illegal mutations impossible during generation
- Authority boundaries enforced at compile time
- No manual authority maintenance

## 10. Workflows (DAG-Based)

### 10.1 Schema (DAG Instead of Procedural Steps)

```yaml
workflows:
  WorkflowName:
    version: "1.0"
    description: "Human-readable description"
    trigger:
      event: EventName
    nodes:
      - name: node_name
        type: command | intent | capability | condition
        target: target_name
        parameters:
          - name: parameter_name
            value: parameter_value
        dependencies:
          - node_name  # must complete before this node
        on_failure: continue | stop | retry
        retry_count: number
```

**DAG Benefits:**
- PING can optimize execution (parallelize independent nodes)
- Automatic retry logic
- Replay support
- Visualization support
- No procedural workflow engine needed

### 10.2 Example: Estimate to Job Workflow (DAG)

```yaml
workflows:
  EstimateToJob:
    version: "1.0"
    description: "Convert accepted estimate to job"
    trigger:
      event: EstimateAccepted
    nodes:
      - name: create_job
        type: command
        target: Job.Create
        parameters:
          - name: estimate_id
            value: ${event.estimate_id}
        dependencies: []
        on_failure: stop
        retry_count: 3
      - name: schedule_appointment
        type: intent
        target: NeedAppointment
        parameters:
          - name: mission_id
            value: ${create_job.job_id}
          - name: constraints
            value:
              start_after: ${event.accepted_at}
              duration: 3600
        dependencies: [create_job]
        on_failure: continue
        retry_count: 0
      - name: notify_crew
        type: capability
        target: Messaging.send_notification
        parameters:
          - name: recipient
            value: ${create_job.crew_id}
          - name: subject
            value: "New Job Assigned"
          - name: body
            value: "You have been assigned to job ${create_job.job_id}"
        dependencies: [create_job]
        on_failure: continue
        retry_count: 3
```

**PING Optimizes:**
- `schedule_appointment` and `notify_crew` can run in parallel (both depend only on `create_job`)
- Automatic retry on failure
- DAG visualization for debugging
- Replay support for audit trails

## 13. Compiler Implementation Order

### 13.1 Fastest Path to Operational Platform

**Order of Implementation:**

1. **Manifest Validator**
   - Validates constitutional schemas
   - Validates references
   - Validates state machines
   - Validates policies
   - Generates validation errors

2. **Manifest → IR Compiler**
   - Normalizes manifest into canonical IR
   - Infers authorities from mutation ownership
   - Generates canonical events from state transitions
   - Generates commands from state transitions
   - Builds workflow DAGs

3. **IR → Event Model Generator**
   - Generates canonical event schemas
   - Generates aggregate definitions
   - Generates transition metadata
   - Generates replay metadata

4. **IR → Repository Generator**
   - Generates event store implementations
   - Generates aggregate repositories
   - Generates snapshot logic
   - Generates projection builders

5. **IR → Capability Runtime**
   - Generates capability operation handlers
   - Generates ABI adapters
   - Generates provider driver interfaces
   - Generates error handling

6. **IR → API & SDK Generator**
   - Generates REST endpoints
   - Generates GraphQL schemas
   - Generates SDKs (TypeScript, Python, Go)
   - Generates OpenAPI documentation
   - Generates tests (unit, integration, replay)

### 13.2 Only After Compiler Exists

**Then Add Business Capabilities:**
- Estimating
- Scheduling
- Invoicing
- CRM

**Every new domain concept is a change to the constitution—not handwritten runtime code.**

## 15. Conclusion

The Generation Manifest is a compiler input that PING transforms into an Intermediate Representation (IR), which then generates 80-90% of the runtime code.

**Key Principles:**
1. **Four Primitives Only:** Identity, Mission, Evidence, Observation
2. **No Explicit Commands:** Generated from state transitions
3. **Capability ABI:** Operations with inputs, outputs, errors, side effects, authorities, replay behavior
4. **Generated Authorities:** Inferred from mutation ownership
5. **Canonical Events:** One event model (Created, Updated, Transitioned, Observed, Verified, Deleted)
6. **Collapsed Observations:** Observation → Assertion → Verified Assertion
7. **DAG Workflows:** Declarative graphs instead of procedural steps
8. **Executable Policies:** requires/before/when/assert rules
9. **IR Layer:** Enables multiple generators, linting, optimization, static analysis, migration generation
10. **Compiler First:** Build compiler before adding business capabilities

**Result:**
- 4-week milestone instead of 6-month roadmap
- Constitutional compliance guaranteed by generation
- Rapid evolution through constitution changes
- Reduced human error
- Consistent code patterns
- Provider independence
- Deterministic replay
- Multi-tenant safety
